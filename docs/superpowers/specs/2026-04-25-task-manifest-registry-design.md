# Task Manifest & Registry Design

**Date:** 2026-04-25
**Status:** Approved

## Problem

TaskLab tasks have no structured record of maturity, run history, or version evidence. The operator has no single place to see which tasks are stale, which have never been run, and which are ready to promote to TaskLib. The runbook currently points to loose `outputs/reports/*.md` files with no consistent schema.

## Solution

Three artifacts working together:

1. **Per-task `manifest.yaml`** — operator-written after each run; source of truth
2. **`tasklab/registry.yaml`** — generated summary of all tasks; consulted before any run
3. **`scripts/build-registry.sh`** — aggregates all manifests into the registry
4. **Updated `running-a-task.md`** — registry becomes step 0, manifest update becomes step 5

---

## 1. Per-task `manifest.yaml`

Location: `tasklab/tasks/<service>/<task>/manifest.yaml`

### Schema

```yaml
maturity: 1        # 0=created, 1=works, 2=multiple runs / improved

runs:
  - date: 2026-04-20
    outcome: success   # success | partial | failed
    notes: "First successful end-to-end run"
    versions:
      stripe_cli: "1.19.4"
      stripe_api: "2023-10-16"
      docs_verified_on: "2026-04-20"

promotion_notes: ""   # filled when nominating for TaskLib
```

### Key decisions

- `versions` is a free-form map — each task names its own keys (`stripe_cli`, `supabase_cli`, `google_wallet_api`, etc.) rather than a fixed schema that would break for most tasks
- `docs_verified_on` is a date — best available signal for website doc freshness, since docs don't carry version numbers
- `maturity` is operator-set — the operator decides when to bump; the script never auto-promotes
- `outcome` is explicit — distinguishes "ran but something was wrong" (`partial`) from fully succeeded
- Full run history stays here; only a summary is rolled into the registry

### Maturity levels

| Level | Meaning |
|-------|---------|
| 0 | Created — task exists but has never been run end-to-end |
| 1 | Works — at least one successful end-to-end run recorded |
| 2 | Hardened — multiple runs, improvements made, stable |

## 2. Central `tasklab/registry.yaml`

Generated by `scripts/build-registry.sh`. **Do not edit by hand.**

### Schema

```yaml
# Generated by build-registry.sh — do not edit by hand
generated: 2026-04-25

tasks:
  - service: stripe
    task: webhooks/setup-and-verify
    maturity: 2
    last_run: 2026-04-25
    last_outcome: success
    run_count: 3
    promotion_notes: ""
    path: tasklab/tasks/stripe/webhooks/setup-and-verify

  - service: apple
    task: wallet-passes/ios-wallet
    maturity: 0
    last_run: null
    last_outcome: null
    run_count: 0
    promotion_notes: ""
    path: tasklab/tasks/apple/wallet-passes/ios-wallet
```

### Key decisions

- Only summary fields in the registry — full run history stays in the per-task manifest
- Tasks without a `manifest.yaml` still appear (discovered from `task.yaml` presence) with `maturity: 0` and nulls — new tasks are visible immediately
- `last_outcome` gives the operator an immediate health signal without opening each task
- Operator reads this to decide what to run next; does not write to it directly

### How to read it

| Signal | Action |
|--------|--------|
| `maturity: 0` | Never run — do a first end-to-end pass |
| `last_outcome: partial` or `failed` | Needs attention before promotion |
| `maturity: 1`, multiple runs | Candidate to bump to maturity 2 |
| `maturity: 2`, `promotion_notes` filled | Ready for TaskLib |

## 3. `scripts/build-registry.sh`

Walks `tasklab/tasks/<service>/<task>/`, reads each `manifest.yaml` if present, writes `tasklab/registry.yaml`.

```bash
#!/usr/bin/env bash
# scripts/build-registry.sh
# Usage: ./scripts/build-registry.sh [--tasks-dir <path>] [--output <path>]
set -euo pipefail

TASKS_DIR="${TASKS_DIR:-tasklab/tasks}"
OUTPUT="${OUTPUT:-tasklab/registry.yaml}"

while [[ $# -gt 0 ]]; do
  case $1 in
    --tasks-dir) TASKS_DIR="$2"; shift 2 ;;
    --output)    OUTPUT="$2";    shift 2 ;;
    *) echo "Unknown flag: $1" >&2; exit 1 ;;
  esac
done

require_command() { command -v "$1" &>/dev/null || { echo "Required: $1" >&2; exit 1; }; }
require_command yq

echo "# tasklab/registry.yaml"                                 >  "$OUTPUT"
echo "# Generated by build-registry.sh — do not edit by hand" >> "$OUTPUT"
echo "generated: $(date +%Y-%m-%d)"                           >> "$OUTPUT"
echo ""                                                        >> "$OUTPUT"
echo "tasks:"                                                  >> "$OUTPUT"

find "$TASKS_DIR" -name "node_modules" -prune -o -name "task.yaml" -print | sort | while read -r task_yaml; do
  task_dir="$(dirname "$task_yaml")"
  manifest="$task_dir/manifest.yaml"

  rel="${task_dir#"$TASKS_DIR/"}"
  service="${rel%%/*}"
  task_slug="${rel#"$service/"}"

  if [[ -f "$manifest" ]]; then
    maturity="$(yq '.maturity // 0'                  "$manifest")"
    last_run="$(yq '.runs[-1].date // "null"'        "$manifest")"
    last_outcome="$(yq '.runs[-1].outcome // "null"' "$manifest")"
    run_count="$(yq '.runs | length'                 "$manifest")"
    promo="$(yq '.promotion_notes // ""'             "$manifest")"
  else
    maturity=0; last_run="null"; last_outcome="null"; run_count=0; promo=""
  fi

  cat >> "$OUTPUT" <<ENTRY
  - service: $service
    task: $task_slug
    maturity: $maturity
    last_run: $last_run
    last_outcome: $last_outcome
    run_count: $run_count
    promotion_notes: "$promo"
    path: $task_dir
ENTRY
done

echo "Registry written to $OUTPUT"
```

### Key decisions

- Requires `yq` — fails closed if missing (consistent with tasklab's preflight-first philosophy)
- Discovers tasks from `task.yaml` presence at any depth, pruning `node_modules` — handles both 2-level (`supabase/setup-project`) and 3-level (`stripe/webhooks/setup-and-verify`) layouts
- Accepts `--tasks-dir` and `--output` flags — consistent with existing script conventions
- Tasks without a manifest still appear so nothing is invisible

## 4. Runbook changes (`running-a-task.md`)

### New step 0 — Consult the registry first

Added before current step 1:

```markdown
## 0) Check the registry

Before picking a task, run:

    ./scripts/build-registry.sh

Then open `tasklab/registry.yaml` to see:

- Tasks at maturity 0 — never run; need a first end-to-end pass
- Tasks with `last_outcome: partial` or `failed` — need attention
- Tasks at maturity 1 with multiple runs — candidates to bump to maturity 2
- Tasks at maturity 2 with `promotion_notes` filled — ready for TaskLib

Pick the task that most needs a run based on this snapshot.
```

### Updated step 5 — Capture evidence + update manifest

Replaces the current loose "capture evidence in outputs/reports/*.md":

```markdown
## 5) Verify success and update the manifest

After a successful run, update (or create) `manifest.yaml` in the task folder:

- Set `maturity` if this run warrants a bump (0→1 on first working run; 1→2 after repeated runs with improvements)
- Append a new entry under `runs` with today's date, outcome, notes, and versions
- Fill `promotion_notes` if the task feels TaskLib-ready

Then regenerate the registry:

    ./scripts/build-registry.sh

Commit both the manifest and the updated registry together.
```

## Out of scope

- Auto-bumping maturity (operator judgment required)
- A TUI or status dashboard (Approach C — premature until manifest schema is proven)
- Manifest validation schema/linting (can be added once the format is stable across a few tasks)
